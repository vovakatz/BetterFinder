# Open in Terminal — Design

## Goal

Add an "Open in Terminal" item to the file browser context menu that launches the system Terminal.app at the appropriate location, with smart behavior depending on what the user right-clicked.

## Behavior

The menu item appears in both the per-item context menu and the empty-space (background) context menu of `FileListView`. Behavior depends on the right-click target:

1. **Empty space** — open Terminal.app with the pane's current folder as cwd.
2. **Folder (non-package)** — open Terminal.app with that folder as cwd.
3. **File (or package)** — open Terminal.app with the file's parent folder as cwd, then type the filename (followed by a single space) into the new Terminal window so the user can hit Enter to execute, or continue typing arguments.

For multi-selection, the action operates only on the right-clicked item, not the full selection. Spawning multiple Terminal windows from one click would be hostile.

The menu item is not shown for network items (`network://`, `smb://`, `afp://` schemes), matching the existing pattern for "Show in Finder" etc.

## Architecture

### New file: `BetterFinder/Services/TerminalLauncherService.swift`

A small stateless service. Two public methods:

```swift
func openTerminal(at folderURL: URL)
func openTerminal(at folderURL: URL, prefilledFilename: String)
```

**`openTerminal(at:)`** — uses `NSWorkspace.shared.open(_:withApplicationAt:configuration:completionHandler:)` to open the folder URL with Terminal.app (`com.apple.Terminal`). Terminal.app natively interprets a directory URL as "open a new window with this as cwd."

**`openTerminal(at:prefilledFilename:)`** — first calls `openTerminal(at:)`, then runs an `NSAppleScript` after a short delay (~250ms) to give Terminal time to create the window and become frontmost. The script is:

```applescript
tell application "System Events"
    tell process "Terminal"
        keystroke "<escaped filename> "
    end tell
end tell
```

The filename is escaped for AppleScript string literal safety (backslashes and quotes). The trailing space lets the user keep typing arguments before hitting Enter.

If the AppleScript fails (e.g., Automation permission denied), the failure is silent — the user still has Terminal open at the right folder, just without the prefill. They can grant permission in System Settings → Privacy & Security → Automation later.

### Modified: `BetterFinder/Views/FileList/FileListView.swift`

Wire the menu items in:

- `backgroundContextMenu` (currently at line 354): add a `Button("Open in Terminal")` that calls `launcher.openTerminal(at: navigationState.currentURL)`.
- `contextMenuContent(for:)` (currently at line 378): add a `Button("Open in Terminal")` inside the existing `if scheme != "network" && scheme != "smb" && scheme != "afp"` guard, placed near "Show in Finder". Behavior:
  - If `displayItem.fileItem.isDirectory && !displayItem.fileItem.isPackage`: call `openTerminal(at: displayItem.fileItem.url)`.
  - Otherwise (regular file, or package treated as a file): call `openTerminal(at: displayItem.fileItem.url.deletingLastPathComponent(), prefilledFilename: displayItem.fileItem.url.lastPathComponent)`.

The launcher is instantiated as a single shared instance (e.g. a `let launcher = TerminalLauncherService()` property on `FileListView`, or accessed statically) — no need for environment injection since it holds no state.

### Modified: `BetterFinder/Info.plist` (or equivalent in `BetterFinderApp.swift`)

Add `NSAppleEventsUsageDescription` so the macOS Automation prompt has a friendly message. Suggested copy:

> "BetterFinder uses System Events to type filenames into Terminal windows when you choose 'Open in Terminal' on a file."

If `Info.plist` is auto-generated by Xcode (modern Swift projects often use the `INFOPLIST_KEY_*` build settings), the equivalent build setting key is `INFOPLIST_KEY_NSAppleEventsUsageDescription`.

## Out of Scope

- **In-app terminal widget integration.** This feature uses system Terminal.app exclusively. The existing `TerminalEmulatorView` / `TerminalSession` are untouched.
- **Other terminal apps (iTerm2, Ghostty, Warp, etc.).** Could be added as a user setting later. Not now.
- **Clipboard fallback when Automation is denied.** Silent failure is fine; the user's recourse is granting the permission once.
- **Acting on multi-selection.** Only the right-clicked item is used.

## Testing Notes

No automated test suite exists in this repo. Manual verification:

1. Right-click empty space in a folder → menu shows "Open in Terminal" → Terminal opens at that folder.
2. Right-click a folder → Terminal opens inside that folder.
3. Right-click a file → Terminal opens at the parent folder, with `<filename> ` already typed at the prompt; pressing Enter executes it.
4. Right-click a file the first time after install → macOS Automation prompt appears for "System Events"; granting it makes the prefill work; denying it leaves Terminal open without prefill (no crash, no error dialog).
5. Right-click a network item (SMB share, etc.) → "Open in Terminal" is not shown.
6. Filenames with spaces, single quotes, double quotes, and backslashes type correctly into Terminal.
